## Домашнее задание к занятию "13.Системы мониторинга" FOPS-38 (Щербатых А.Е.)

### Обязательные задания

1. Вас пригласили настроить мониторинг на проект. На онбординге вам рассказали, что проект представляет из себя платформу для вычислений с выдачей текстовых отчетов, которые сохраняются на диск. Взаимодействие с платформой осуществляется по протоколу http. Также вам отметили, что вычисления загружают ЦПУ. Какой минимальный набор метрик вы выведите в мониторинг и почему?

**Ответ**

1. Метрики доступности и работоспособности HTTP

Платформа взаимодействует по HTTP → первым делом нужно понимать, отвечает ли она и насколько быстро.

- ```http_requests_total``` (счётчик) — общее количество запросов. Важные лейблы: method, path, status_code (или группа 2xx/3xx/4xx/5xx).
- ```http_request_duration_seconds``` (гистограмма) — время выполнения запросов (латентность). Позволит строить перцентили (p50, p95, p99).

Зачем:

Если платформа не отвечает на HTTP или отвечает ошибками 5xx — бесполезно смотреть на CPU. Если латентность резко выросла, это первый звоночек проблем с вычислениями или диском.

2. Метрики загрузки CPU (ключевое по условию)

Вычисления грузят процессор — значит, нужно отличать нормальную утилизацию от перегрузки.

- ```cpu_usage_seconds_total``` (счётчик) — удобно через rate() смотреть среднюю загрузку за интервал. Лучше в долях ядра или процентах.
- ```cpu_load_average_1m/5m/15m``` (в разрезе ядер) — чтобы видеть, не накапливаются ли задачи в очереди.

Зачем:

Если CPU зашкаливает (>80–90% долгое время) — вычисления не успевают, латентность HTTP растёт, отчёты могут задерживаться. 

3. Метрики дисковых операций (сохранение отчётов)

Отчёты сохраняются на диск. Проблемы с диском напрямую влияют на возможность сохранить результат.

- ```disk_write_operations_total``` (счётчик) и disk_write_bytes_total
- ```disk_io_time_seconds_total``` или ```disk_await``` (среднее время ожидания операции) — самый важный показатель.

Зачем:

Высокий CPU может создавать много мелких записей на диск. Если диск не успевает (высокий await), то сохранение отчёта тормозит → HTTP запрос висит → пользователь не получает отчёт. Без метрик диска можно будет подумать, что проблема в CPU, а на самом деле — в дисковой подсистеме.

4. Метрики ошибок сохранения (логика приложения)

HTTP может отвечать 200, а отчёт не сохраниться (диск полон, нет прав, сбой при записи).

```report_generation_errors_total``` (счётчик) — кастомная метрика из кода платформы. Инкрементится при любой неудаче сохранения отчёта.

Зачем:

Это критично для бизнес-логики. Без неё мониторинг покажет «всё зелёное», а отчёты не создаются.
